# ファイルサーバーハンズオン

## 目的
Windowsファイルサーバーに共有フォルダを作成し、Active Directoryのグループを利用したアクセス制御を学習する。
ユーザーへ直接権限付与するのではなく、グループを経由して権限を管理する運用方法を理解することを目的とする。

## フォルダ構成
FILE01に共有フォルダ作成。
C:\Share
  Sales
共有名：Sales

## 共有権限
Evevyone
フルコントロール
理由
実際のアクセス制御はNTFS権限で実施するため、共有権限はシンプルな設定とした。
共有権限とNTFs権限が両方設定されている場合、より厳しい権限が適用されることを学習した。

## NTFS権限
SalesフォルダのNTFS権限を設定

### 最終構成
SYSTEM          フルコントロール
Administrators  フルコントロール
GG_Sales        変更
理由
一般ユーザーには業務に必要な操作のみ許可し、権限変更や所有者変更は管理者のみ実施できるようにするため。
最小権限の原則に基づき、GG_Salesには「変更」権限を付与した。

## 継承
Salesフォルダ作成直後は親フォルダから権限が継承されていた。
そのため、Usersグループを削除しようとした際にエラーが発生。
対応手順
Sales → プロパティ → セキュリティ → 詳細設定 → 継承の無効化 → 継承されたアクセス許可を明示的なアクセス許可へ変換
その後、不要な権限を削除

## 学んだこと
Windowsのアクセス権は親フォルダから子フォルダへ継承される。
共有フォルダごとに異なる権限を設定したい場合は、軽傷を無効化して個別に管理する


## グループによる権限管理
ユーザーへ直接権限を付与するのではなく、グループへ権限を付与した。
構成
yamada → GG_Sales → Salesフォルダ
理由
移動や退職が発生した場合でも、グループの所属変更のみで権限管理ができるため

### 動作確認

### アクセス許可
ユーザー：yamada
条件：GG_Sales
結果：\\FILE01\Salesアクセス成功

### アクセス拒否
ユーザー：test02
条件：GG_Sales未所属
結果：アクセス拒否

### 学んだこと
- 共有権限とNTFS権限は両方評価され、より厳しい権限が適用される
- ユーザーではなくグループへ権限を付与する
- Windowsの権限は継承される
- 必要に応じて継承を無効化する
- 最小権限の原則に基づき権限を設定する
- ファイルサーバーの権限管理は「ユーザー → グループ → 権限」の構成で管理する

## アクセス監査

### 目的
共有フォルダへのアクセスを記録し、
- 誰がアクセスしたか
- いつアクセスしたか
- どのファイルにアクセスしたか
を追跡できるようにする。実務では情報漏洩調査やファイル削除調査などで利用される。

### 監査ポリシーの有効化
FILE01でローカルセキュリティーポリシーを設定
設定項目
ローカルセキュリティポリシー → ローカルポリシー → 監査ポリシー → オブジェクトアクセスの監査
設定
成功、失敗 両方有効化

### auditpolによる監査設定
PowerShell管理者権限で実行
auditpol /set /subcategory:"ファイル システム" /success:enable /failure:enable
確認
auditpol /get /subcategory:"ファイル システム"
結果
ファイル システム
成功及び失敗

### 監査エントリの設定
対象フォルダ
C:\Shares\Sales
設定
Sales → プロパティ → セキュリティ → 詳細設定 → 監査
監査対象：Everyone
監査内容：読み取り、フォルダーの内容一覧表示、読み取りと実行

### 動作確認
CLIEN01から\\FOLE01\Salesへアクセス
その後FILE01でイベントビューアーを確認
イベントビューアー → WIんどｗｓログ → セキュリティ
イベントID：4663
を確認
確認結果
subjectUserName : yamada
subjectDomainNmae : CORP
ObjectName : C:\Shares\Sales

### 学んだこと
監査設定を行うことで、だれが、いつ、どこへ、アクセスしたかを記録できる。
権限設定（DACL）と監査設定（SACL）は別物
DACL：アクセス許可・拒否
SACL：アクセス記録

## ファイル削除監査

### 目的
ファイル削除時の操作ログを取得し、削除したユーザーを特定できるようにする

### 設定
監査エントリに削除監査を追加
対象：Everyone
監査：削除（Delete）

### 動作確認
CLIENT01で\\FILE01\Salesへアクセス
テストファイルを作成 → test.txt
その後削除

### イベントログ確認
FILE01でイベントビューアー確認
イベントID：4663
確認結果
ObjectNmae： C:\\Shares\Sales\test.txt
AccessList: %%1537

### 結果
%%1537は削除操作（DELETE）を表す。
これにより、誰が、どのファイルを、削除したかを特定できることを確認
